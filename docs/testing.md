# Testing

Mactrix uses snapshot and layout tests to catch visual regressions and layout stability issues across the UI. Tests live in `MactrixLibrary/Tests/UITests/` and run via Swift Package Manager.

## Running Tests

```bash
cd MactrixLibrary
swift test --filter UITests
```

The full suite (~30 tests) runs in under a second.

## Test Structure

| File | Purpose |
|---|---|
| `SnapshotTestHelper.swift` | Shared helper for rendering SwiftUI views into `NSHostingView` and asserting snapshots |
| `PreviewSnapshotTests.swift` | Snapshot tests covering all MactrixLibrary UI views, grouped by area (timeline, sidebar, inspector, rooms, shared components) |
| `MessageEventBodyViewSnapshotTests.swift` | Targeted snapshot tests for the message event body in different states (reactions, read receipts, empty) |
| `MessageEventBodyViewLayoutTests.swift` | Layout stability tests — verifies view tree behaviour and guards against infinite layout loops |
| `__Snapshots__/` | Reference PNG images committed to the repository |

## Snapshot Tests

Snapshot tests render each view into an `NSHostingView`, capture a PNG, and compare it against a committed reference image. They use [swift-snapshot-testing](https://github.com/pointfreeco/swift-snapshot-testing) with a perceptual precision tolerance of 98% to account for minor rendering differences.

### First run on a new machine

On first run (or after deleting `__Snapshots__/`), all snapshot tests will fail and record new reference images. Run the tests a second time to verify they pass against the newly recorded references.

### Updating snapshots after intentional UI changes

If you've changed a view's appearance intentionally:

1. Delete the relevant snapshot file(s) from `__Snapshots__/`
2. Run `swift test --filter UITests` to record new references
3. Run again to confirm they pass
4. Commit the updated PNGs alongside your code changes

Alternatively, temporarily set the record mode in `SnapshotTestHelper.swift` to `.all` to re-record everything, then set it back to `.missing` before committing.

## Layout Tests

Layout tests verify structural behaviour rather than visual appearance:

- **`bottomRowCollapsesWithoutContent`** — Asserts that the message reaction/read receipt row adds height when content is present and collapses when empty.
- **`rapidReactionToggleDoesNotHang`** — Renders the message view 20 times with random reaction states under a time limit, catching infinite layout loop regressions caused by view tree instability in `NSHostingView`.

Both tests have a 1-minute timeout. If a layout loop occurs, the test hangs and fails on timeout.

## CI: Snapshot Generation and Verification

Snapshot images are pixel-sensitive to macOS version, display scale, and font rendering. To ensure consistency, reference snapshots should be generated by CI rather than committed from developer machines. The recommended approach is for CI to record any missing snapshots and commit them back to the PR branch, so that:

- Reference images are always generated in a controlled environment
- Visual changes are reviewable as PNG diffs in pull requests
- Developers don't need to worry about local rendering differences

### How it works

1. A developer changes a view and deletes the corresponding snapshot file(s) from `__Snapshots__/`
2. They push the commit (with the deleted PNGs) to their PR branch
3. CI detects the missing snapshots, records them, and commits the new PNGs back to the branch
4. The developer (and reviewers) can inspect the updated snapshots in the PR diff
5. On subsequent pushes where snapshots haven't changed, CI simply verifies against the committed references

### Workflow implementation

The following GitHub Actions workflow implements this approach. Key points:

- **Pin the macOS runner version** (e.g. `macos-15`) — never use `macos-latest`, as runner updates would invalidate all snapshots
- **Pin the Xcode version** — font rendering and view layout can change between Xcode releases
- **Use `[ci skip]` in the commit message** to prevent the snapshot commit from triggering another CI run

```yaml
name: UI Snapshot Tests
on: [pull_request]

jobs:
  snapshot-tests:
    runs-on: macos-15
    permissions:
      contents: write
    steps:
      - uses: actions/checkout@v4
        with:
          ref: ${{ github.head_ref }}
          token: ${{ secrets.GITHUB_TOKEN }}

      - name: Select Xcode
        run: sudo xcode-select -s /Applications/Xcode_16.2.app

      - name: Run snapshot tests
        id: first_run
        continue-on-error: true
        run: cd MactrixLibrary && swift test --filter UITests

      - name: Record and commit missing snapshots
        if: steps.first_run.outcome == 'failure'
        run: |
          # Run again to record any missing snapshots
          cd MactrixLibrary && swift test --filter UITests || true
          cd ..

          # Commit new/updated snapshot PNGs if any were generated
          git add MactrixLibrary/Tests/UITests/__Snapshots__/
          if git diff --cached --quiet; then
            echo "No new snapshots to commit"
            exit 1  # Tests failed for a reason other than missing snapshots
          fi

          git config user.name "github-actions[bot]"
          git config user.email "github-actions[bot]@users.noreply.github.com"
          git commit -m "test: update snapshot references [ci skip]"
          git push

      - name: Verify snapshots
        if: steps.first_run.outcome == 'failure'
        run: cd MactrixLibrary && swift test --filter UITests
```

### Workflow behaviour

| Scenario | What happens |
|---|---|
| No view changes, snapshots up to date | First run passes, job succeeds |
| View changed, old snapshot deleted | First run fails (missing reference), records new PNG, commits it, verify step passes |
| View changed, snapshot not deleted | First run fails (pixel mismatch), no new files to commit, job fails — developer must delete the stale snapshot |
| New view added with new test | First run fails (missing reference), records new PNG, commits it, verify step passes |
| Layout test regression | First run fails, no snapshots to commit, job fails |

### Local development

Developers can run snapshot tests locally for quick feedback, but should not commit locally-recorded PNGs to the repository. If local snapshots differ from CI-generated ones (due to macOS version differences), delete the local `__Snapshots__/` directory and let CI re-record them.

To preview snapshots locally without committing:

```bash
cd MactrixLibrary
rm -rf Tests/UITests/__Snapshots__
swift test --filter UITests  # Records references
swift test --filter UITests  # Verifies against them
git checkout -- Tests/UITests/__Snapshots__  # Restore CI-generated references
```

## Adding Tests for New Views

When adding a new view to `MactrixLibrary/Sources/UI/`:

1. Add a `#Preview` block to the view file with representative mock data
2. Add a corresponding snapshot test in `PreviewSnapshotTests.swift` under the appropriate section
3. Use `SnapshotTestHelper.assertViewSnapshot()` with a suitable width (600pt default, or narrower for sidebar/inspector views)
4. Push to your PR branch — CI will record the reference snapshot and commit it
5. Review the generated PNG in the PR diff

For views with complex layout behaviour (conditional content, dynamic sizing, AppKit embedding), consider adding a targeted layout test as well.

## Dependencies

The test target depends on:

- [swift-snapshot-testing](https://github.com/pointfreeco/swift-snapshot-testing) (v1.17+) — snapshot assertion library
- `UI` module — the views under test
- `Models` module (transitive) — mock types used in test fixtures
